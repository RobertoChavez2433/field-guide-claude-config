# Pattern: Sync Adapter via AdapterConfig

## How We Do It

The sync engine uses a data-driven approach: 13+ simple tables are declared as `AdapterConfig` entries in `lib/features/sync/adapters/simple_adapters.dart`. Each entry produces a `TableAdapter` via `AdapterConfig.toAdapter()` (delegating to `_ConfiguredAdapter`). FK dependencies, scope type (direct / viaProject / viaEntry), converters, natural keys, and file-backed behavior are all declarative. Complex adapters still subclass `TableAdapter` directly (e.g. `InspectorFormAdapter`), but most tables fit the config model.

For **file-backed** tables (signatures, photos, entry_exports, form_exports, export_artifacts), the config sets `isFileAdapter: true`, a `storageBucket`, a `localFilePathColumn`, and a `buildStoragePath` function that shapes the remote storage path.

## Exemplar — Standard Adapter (entry_contractors)

```dart
AdapterConfig(
  table: 'entry_contractors',
  scope: ScopeType.viaEntry,
  fkDeps: ['daily_entries', 'contractors'],
  fkColumnMap: {'daily_entries': 'entry_id', 'contractors': 'contractor_id'},
  naturalKeyColumns: ['entry_id', 'contractor_id'],
),
```

## Exemplar — File Adapter (form_exports)

```dart
AdapterConfig(
  table: 'form_exports',
  scope: ScopeType.viaProject,
  fkDeps: ['projects', 'form_responses'],
  fkColumnMap: {
    'projects': 'project_id',
    'form_responses': 'form_response_id',
  },
  localOnlyColumns: ['file_path'],
  isFileAdapter: true,
  storageBucket: 'form-exports',
  stripExifGps: false,
  buildStoragePath: _buildFormExportPath,
  extractRecordName: _extractExportRecordName,
),
```

## Reusable Methods

| Method | File:Line | Signature | When to Use |
|---|---|---|---|
| `AdapterConfig` constructor | `adapter_config.dart:80` | all-named-params | Declare table adapter |
| `_ConfiguredAdapter` | `adapter_config.dart:112` | private wrapper | Auto-generated by `toAdapter()` |
| `FileSyncHandler` | `file_sync_handler.dart:26` | three-phase upload | Called by PushHandler for `isFileAdapter=true` |
| `SyncRegistry.adapterFor(tableName)` | `sync_registry.dart:94` | lookup | Runtime adapter resolution |

## 1126 Application

Add two entries to `simpleAdapters`:

```dart
// Standard adapter — metadata only, no file payload
AdapterConfig(
  table: 'signature_audit_log',
  scope: ScopeType.viaProject,
  fkDeps: ['projects', 'signature_files'],
  fkColumnMap: {
    'projects': 'project_id',
    'signature_files': 'signature_file_id',
  },
),

// File-backed adapter — PNG upload to dedicated bucket
AdapterConfig(
  table: 'signature_files',
  scope: ScopeType.viaProject,
  fkDeps: ['projects'],
  fkColumnMap: {'projects': 'project_id'},
  localOnlyColumns: ['local_path'],
  isFileAdapter: true,
  storageBucket: 'signatures',
  localFilePathColumn: 'local_path',
  stripExifGps: false,   // PNG has no EXIF; explicit for clarity
  buildStoragePath: _buildSignatureFilePath,
),

String _buildSignatureFilePath(String companyId, Map<String, dynamic> record) {
  // Matches the existing pattern from _buildFormExportPath
  final projectId = record['project_id'];
  final id = record['id'];
  return '$companyId/$projectId/$id.png';
}
```

## FK Ordering

`signature_audit_log` depends on `signature_files` → so `signature_files` MUST appear earlier in `simpleAdapters` (or be listed first in `fkDeps`). Also, audit log depends on `projects` and — because it references `form_response` via `signed_record_id` but that is a soft foreign key (string column, not a SQL FK) — we do NOT list `form_responses` in `fkDeps`. The sync engine only enforces listed deps.
